## Patterns Observed

### 1. Template artifact residue (5 of 11 unclassified candidates)

Every finding below traces directly to the cookiecutter scaffolding step, not to developer drift. The template emitted these problems into a freshly generated repo:

- `FOUND-CHANGELOG-001`: CHANGELOG references Poetry instead of UV (1 repo; systematic in all UV-based template outputs)
- `FOUND-PKG-001`: `pyproject.toml` missing `project.urls` and has empty `keywords` (confirmed in 3+ cookiecutter-python-template scaffolds)
- `FOUND-PKG-002`: Incomplete Python version classifiers in `pyproject.toml` (1 repo; template gap)
- `FOUND-README-002`: README footer ends with an incomplete sentence (1 repo; template boilerplate defect)
- `FOUND-README-003`: Badge URLs use lowercase repo name; actual repo uses mixed case (1 repo; template variable interpolation gap)

**Root cause:** The cookiecutter template does not drive `project.urls`, `keywords`, Python classifiers, or badge URLs from context variables at generation time. The CHANGELOG boilerplate still names Poetry as the package manager.

**Template feedback filed:** All five items added to `docs/template_feedback.md`.

### 2. qlty silent-skip wrapping (3 checks: PC-003, CI-007b, PC-005)

The scaffolded `.pre-commit-config.yaml` and associated qlty integration wrapped `basedpyright`, `ruff`, and several linting hooks in configurations that caused qlty to silently skip rather than execute them. The pattern is:

```yaml
# Problematic pattern
- repo: local
  hooks:
    - id: basedpyright
      entry: bash -c 'basedpyright "$@" || true'  # silent pass-through
```

When a hook is wrapped in `|| true` or otherwise made to always exit 0, the pre-commit gate becomes cosmetic. The scaffold's qlty integration was the source: qlty's shim layer generates silent-skip behavior for hooks it does not natively support rather than failing audibly.

**Impact:** PC-003 (basedpyright absent/silent), PC-005 (ruff silent), and CI-007b (hook name mismatch) were all remediated in this session. Any repo generated from this template before the fix carries the same exposure.

**Template feedback filed.** The qlty integration config needs explicit `fail_fast: true` and a documented fallback policy.

### 3. CI-022/023: Required status check names do not match workflow producer names (deferred)

The org branch-protection ruleset specifies required status check context strings (`Security Analysis / Security Gate Validation`, `Check REUSE Compliance`) that do not match the `name:` fields of the workflow jobs that are supposed to satisfy them. This means the ruleset does not actually gate merges on those workflows for any freshly scaffolded repo until the names are manually reconciled.

**Pattern:** Every repo generated from this template and added to the org before the ruleset context strings are corrected will have an effectively unenforced security gate. This is a systemic scaffolding + org-config alignment gap, not a per-repo quirk.

**Status:** Deferred (CI-022, CI-023). Requires coordinated change to either the workflow `name:` fields or the org ruleset context strings. The deferred items are tracked in PR #3.

### 4. REUSE.toml merge not performed at scaffold time (FOUND-REUSE-002)

The template generates a `.standards/REUSE.baseline.toml` file but does not merge it into `REUSE.toml` automatically. The merge step is documented as a manual developer action. As a result, newly scaffolded repos have no SPDX headers in `src/`. REUSE CI fails until the developer runs the merge.

**Unclassified candidate:** `FOUND-REUSE-002` (No SPDX headers in `src/`).

### 5. Noxfile stub sessions advertised but not implemented (FOUND-NOX-001, FOUND-NOX-002)

The scaffolded `noxfile.py` declares `perf` and `mutate` sessions that raise `NotImplementedError`, and does not include `bandit` or `pip-audit` sessions. Running `nox -l` presents these as available, creating false confidence.

**Pattern:** Confirmed in 3+ repos generated from this template.

### 6. Deprecated `version:` key in Compose files (FOUND-DOCKER-002)

The scaffolded `docker-compose*.yml` files carry the top-level `version:` key, which Docker Compose v2 ignores with a deprecation warning. All generated repos exhibit this.

**Pattern:** Confirmed in 3+ repos generated from this template.

### 7. known-vulnerabilities.md location mismatch (FOUND-OSSF-002 variant)

The CLAUDE.md standard references `docs/known-vulnerabilities.md` but the template places the file at the project root (`known-vulnerabilities.md`). Any audit script or CI check that looks for the canonical path will report a false absence.
