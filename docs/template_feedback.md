### `.markdownlint.json` trips on multi-phase planning docs

- **Priority**: Low
- **Category**: Configuration
- **Discovered**: 2026-06-01

**Issue**: The shipped `.markdownlint.json` leaves `MD024` (no-duplicate-heading)
at its default and does not address `MD060` (table-column-style). The planning
documents produced by the `/plan` workflow (roadmap, ADRs) intentionally repeat
sibling subheadings such as `### Objective` and `### Deliverables` under each
phase, and use standard GFM tables. This generates dozens of editor warnings on
freshly generated, valid docs.

**Context**: Discovered while generating `docs/planning/` documents via `/plan`.
Confirmed markdownlint is not part of the enforced gate (no pre-commit hook, not
enabled in qlty), so the warnings are IDE-only but still noisy.

**Suggested Fix**: Ship `"MD024": { "siblings_only": true }` and `"MD060": false`
in the template's `.markdownlint.json`, so the planning/ADR templates lint clean
out of the box.

**Affected Files**: `.markdownlint.json`

### `.claude/settings.json` Bash permission colon syntax is wrong

- **Priority**: High
- **Category**: Configuration
- **Discovered**: 2026-06-01

**Issue**: The template generates `"Bash(cmd:*)"` entries in `.claude/settings.json`
permission allow lists. Colon syntax is the MCP tool format; shell commands require
space syntax `"Bash(cmd *)"`. All generated entries are silently inert at runtime,
meaning every shell command still prompts for permission despite being listed.

**Context**: Discovered during compliance audit PR #3 review via `pr-review` skill.
All 6 generated entries used colon syntax and were flagged as a Critical finding.

**Suggested Fix**: Change the template's `.claude/settings.json` allow entries from
`"Bash(cmd:*)"` to `"Bash(cmd *)"` (space separator, not colon).

**Affected Files**: `.claude/settings.json`

### `mkdocs.yml` `site_description` not quoted, breaks YAML parse

- **Priority**: High
- **Category**: Configuration
- **Discovered**: 2026-06-01

**Issue**: The generated `mkdocs.yml` has `site_description` with an unquoted value
containing a colon-space sequence (`assistant: RAG over...`). YAML interprets the
unquoted colon-space as a mapping separator, producing a parse error at build time:
"mapping values are not allowed here, line 6, column 83".

**Context**: Discovered when the Documentation CI check failed with a YAML parse
error. The mkdocs.yml shipped by the template was the immediate cause.

**Suggested Fix**: Wrap the default `site_description` value in double quotes in
the template, or use a value that doesn't contain colons.

**Affected Files**: `mkdocs.yml`

### `no-em-dash` pre-commit hook self-flags its own config file

- **Priority**: Medium
- **Category**: Tooling
- **Discovered**: 2026-06-01

**Issue**: The `no-em-dash` hook added by the compliance system uses `entry: "—"`
(the em-dash character itself). Because the hook's own definition in
`.pre-commit-config.yaml` contains this character, the hook flags its own config
file on every run unless `.pre-commit-config.yaml` is in the exclude list.

**Context**: Discovered during pr-fix session for compliance PR #3. Running
`pre-commit run --all-files` would have failed on `.pre-commit-config.yaml`.

**Suggested Fix**: Add `\.pre-commit-config\.yaml` to the no-em-dash hook's
`exclude` pattern. Also check that any CLAUDE.md generated by the template that
documents the em-dash rule uses `(U+2014)` instead of the literal em-dash character.

**Affected Files**: `.pre-commit-config.yaml`

### `pr-validation.yml` references deprecated org workflow `python-pr-validation.yml`

- **Priority**: High
- **Category**: CI/CD
- **Discovered**: 2026-06-01

**Issue**: The template generates a `pr-validation.yml` with a `core-validation` job
that calls `python-pr-validation.yml` from the org `.github` repo. That workflow has
been deprecated and replaced with a migration stub that intentionally exits 1 with
the message "Migrate to python-ci.yml". The SHA pinning by the compliance system
locked in the deprecated commit, causing every PR to fail with "Core Validation /
This workflow is removed".

Additionally, `python-ci.yml` (the replacement) uses `--no-build` during install,
which is incompatible with editable installs. The `core-validation` call provides
duplicate value since `ci.yml` already runs Python tests and quality checks.

**Context**: Discovered during compliance audit PR #3 review and subsequent pr-fix.
Two fix attempts were needed: first migrating to `python-ci.yml`, then removing
the job entirely when the install compatibility issue surfaced.

**Suggested Fix**: Remove the `core-validation` reusable workflow call from the
template's `pr-validation.yml`. Keep it focused on PR-specific checks (changelog,
dead code, link check). Python quality checks belong in `ci.yml`.

**Affected Files**: `.github/workflows/pr-validation.yml`

### `validate-front-matter` hook scans the whole `docs/` tree, so latent failures block unrelated commits

- **Priority**: High

**Issue**: The `validate-front-matter` pre-commit hook sets `pass_filenames: false`
and runs `python tools/validate_front_matter.py docs`, so it re-validates the entire
`docs/` tree on every trigger instead of only the staged files. The shipped
`docs/ADRs/adr-template.md` (`schema_type: adr`) and the brainstorming spec under
`docs/superpowers/specs/` (no front matter) are both non-compliant with the validator
out of the box, but the failure stays latent because the hook only triggers on a
non-`planning` `docs/*.md` change. The first unrelated doc commit surfaces all of
them at once and is blocked by files the author never touched.

**Context**: Committing two new docs (a compliance report and template feedback)
failed the hook with errors pointing at `adr-template.md` and the superpowers spec,
neither of which was staged.

**Suggested Fix**: Either honor staged filenames (`pass_filenames: true` and validate
only the passed paths), or ship the template's own `docs/` files already compliant
with the validator. A hook should not fail on files outside the current change set.

**Affected Files**: `.pre-commit-config.yaml`, `tools/validate_front_matter.py`,
`docs/ADRs/adr-template.md`, `docs/superpowers/specs/*.md`

### Front-matter schema lacks ADR statuses and a report/lessons-learned doc type

- **Priority**: Medium

**Issue**: The discriminated-union schema in `tools/frontmatter_contract/models.py`
allows only `status` values `draft`, `in-review`, `published`, and `schema_type`
values `script`, `knowledge`, `planning`, `common`. The shipped `adr-template.md`
uses `status: proposed` (a standard ADR lifecycle state) and `schema_type: adr`,
both rejected by the schema. Separately, `CommonFM` sets `ConfigDict(extra="forbid")`,
so generated report artifacts (for example compliance `lessons-learned` files carrying
`session_date`, `repo`, `manifest_version`) cannot be made compliant without stripping
their metadata into the body.

**Context**: Found while making `adr-template.md`, a compliance retrospective, and a
brainstorming spec pass the validator. The ADR template ships failing its own schema.

**Suggested Fix**: Add ADR lifecycle states (`proposed`, `accepted`, `superseded`,
`deprecated`) to the `status` literal, add an `adr` (or `report`) `schema_type` variant,
and either relax `extra="forbid"` for report types or exclude generated report trees
(like `docs/planning/` is already excluded).

**Affected Files**: `tools/frontmatter_contract/models.py`, `docs/ADRs/adr-template.md`
